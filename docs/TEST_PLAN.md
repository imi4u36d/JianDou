# AI Cut 首版测试计划

## 1. 测试目标

验证 AI 短剧剪辑平台的首版主链路是否稳定可用，重点覆盖以下能力：

- 视频上传
- 任务创建与状态流转
- 按参数生成素材
- 生成数量限制
- 时长范围控制
- 片头片尾拼接
- 失败重试
- 配置文件校验
- 本地环境依赖可用性

本计划以本地开发环境和联调环境为主，目标是确保“能上传、能生成、能回传结果、能导出素材”，并且输出结果满足最基本的投放要求。

## 2. 测试范围

### 2.1 包含范围

- 上传单个视频文件
- 创建短剧剪辑任务
- 查看任务状态、进度和结果
- 根据参数生成 1 条或多条素材
- 结果素材的时长、数量、拼接效果校验
- 失败后的重试、恢复和错误展示
- `config/app.toml` 与 `config/app.example.toml` 的基础配置校验
- 本地依赖项可用性检查

### 2.2 不包含范围

- 多租户权限体系
- 复杂投放平台自动分发
- 高级镜头识别、剧情理解、字幕精修等增强能力
- 大规模分布式压测
- 线上灰度和 A/B 策略

## 3. 测试分层策略

### 3.1 单元测试

覆盖纯逻辑，不依赖外部服务：

- 参数合法性判断
- 任务状态机转移
- 生成数量上限计算
- 时长区间判定
- 片头片尾模板选择逻辑
- 重试次数与退避策略
- 配置项解析与默认值回填

### 3.2 接口测试

覆盖核心 API 的入参、出参和错误码：

- 上传接口
- 任务创建接口
- 任务详情接口
- 任务列表接口
- 结果下载接口
- 重试接口
- 配置读取接口

### 3.3 流程测试

覆盖端到端业务链路：

1. 上传视频
2. 配置剪辑参数
3. 创建任务
4. 执行生成
5. 轮询或查询状态
6. 获取结果素材
7. 预览或下载

### 3.4 媒体专项测试

针对最终成片做质量确认：

- 画面时长
- 音画同步
- 片头片尾拼接
- 黑帧和静帧
- 输出分辨率和比例
- 文件可播放性

### 3.5 非功能测试

- 并发和稳定性
- 失败恢复
- 长任务运行
- 本地资源占用
- 依赖服务不可用时的降级表现

### 3.6 日志链路测试

覆盖任务从上传到输出的事件记录和查询能力：

- 事件完整性：上传、创建任务、状态切换、probe、planning、render、retry、failure 等关键节点都应产生日志事件。
- 阶段顺序：事件时间线应符合实际执行顺序，不能出现渲染早于规划、失败早于开始之类的乱序记录。
- 失败日志：模型失败、转码失败、存储失败、参数校验失败时，日志应包含明确阶段、错误摘要和可追踪详情。
- 重试日志：任务重试后应保留历史事件，并新增重试事件，不能覆盖原始失败上下文。
- 接口返回：`GET /api/v1/tasks/{task_id}/events` 应返回稳定结构，包含时间戳、级别、阶段、消息和详情字段。

## 4. 关键验收用例

### 4.1 上传

- 上传合法视频文件后，系统返回可追踪的文件标识和任务入口。
- 上传空文件、损坏文件、超大文件、格式不支持文件时，系统拒绝并给出明确错误信息。
- 重复上传同一文件时，系统行为一致，不产生脏数据。

### 4.2 任务状态

- 新建任务后状态应从 `pending` 或 `queued` 开始。
- 任务执行中状态应能被前端正确展示。
- 成功后应落到完成态，并可查询到素材结果。
- 失败后应保留失败原因，便于重试和排查。
- 取消任务后应停止继续生成或明确标记为取消态。

### 4.3 生成数量

- 当参数要求生成 `N` 条素材时，最终结果不得超过 `N`。
- 若中途有单条素材失败，系统应明确是跳过、补生成还是整体失败。
- 重复触发同一任务时，不应额外超发素材。

### 4.4 时长范围

- 输出素材时长必须落在配置的最小时长和最大时长之间。
- 边界值要单独验证，例如等于最小值、等于最大值。
- 超出范围时，系统应拒绝生成或在结果中标记不合规。

### 4.5 片头片尾拼接

- 指定片头模板后，输出素材应正确拼接片头，不出现跳帧、黑帧或静音异常。
- 指定片尾模板后，结尾应完整落入输出文件中，不应被裁断。
- 片头、正文、片尾的切点应符合规则，不出现明显错位。

### 4.6 失败重试

- 模型调用失败时，任务应进入可重试状态或失败态。
- 转码失败时，系统应记录失败节点和错误信息。
- 网络超时、存储失败、队列消费失败时，应按配置次数重试。
- 超过最大重试次数后，任务应稳定失败，不进入无限重试循环。

### 4.7 配置文件校验

- `config/app.toml` 中的数据库、Redis、存储目录、模型、pipeline 配置必须能被正确读取。
- 缺失必填项时，启动应直接失败并提示缺失项。
- 非法 URL、错误端口、非法数值、空目录路径应被识别出来。
- `app.example.toml` 与实际配置字段应保持一致，避免遗漏字段。

### 4.8 本地环境依赖

- 后端 API 启动所需的 Python 运行环境可用。
- 前端开发所需的 Node.js / npm 环境可用。
- MySQL 可连接并可创建业务库。
- Redis 可连接并可写入缓存数据。
- FFmpeg 可执行文件可用，且能完成裁切与拼接。
- 本地存储目录 `storage/uploads`、`storage/outputs`、`storage/temp` 可读写。
- 模型服务未配置时，系统应能给出明确提示；如支持离线/模拟模式，则应能在测试环境中替代真实调用。

### 4.9 日志链路追踪

- 创建任务后应能查询到至少一条创建事件和一条调度事件。
- 任务执行期间应能按阶段查到连续事件，不应出现长时间无事件但状态已变化的情况。
- 任务失败后应能查到失败阶段、失败原因和失败详情，且详情页可回显。
- 任务重试后应同时保留首次失败事件和重试后的新事件，便于定位历史问题。
- 日志接口在任务成功、失败、处理中三种状态下都应返回合法结构，不应因空日志或部分日志报错。

## 5. 媒体处理质量校验点

### 5.1 基础校验

- 输出文件可正常播放
- 无明显花屏、黑屏、卡顿
- 无音画不同步
- 无中间文件泄露到最终结果

### 5.2 结构校验

- 素材片段顺序正确
- 切点符合参数和策略
- 片头、正文、片尾完整衔接
- 目标比例和分辨率符合配置

### 5.3 结果校验

- 素材数量符合要求
- 每条素材时长符合要求
- 文件命名和归档规则一致
- 可被前端预览和下载

## 6. 性能、稳定性与失败恢复

### 6.1 性能测试

- 单任务生成耗时在可接受范围内
- 多任务并发提交时系统可稳定排队
- 文件上传和结果下载不会明显阻塞主流程

### 6.2 稳定性测试

- 服务重启后，正在执行的任务能恢复查询或重新进入可处理状态
- 长时间运行后不出现任务卡死
- 资源占用不会持续增长到失控

### 6.3 失败恢复测试

- 模型接口超时后能重试并保留失败记录
- 转码节点失败后可重试或明确失败
- Redis、MySQL、存储目录不可用时能给出可理解的错误提示
- 重复提交和重复回调不应破坏任务最终状态

## 7. 本地环境依赖清单

### 7.1 必需依赖

- Node.js 14.18+，用于前端开发和构建
- npm，用于安装和运行前端脚本
- Python 3.11+，用于后端 API 和 worker
- MySQL 8.x，本地业务库
- Redis 6.x+，任务队列或缓存依赖
- FFmpeg，可执行命令在 PATH 中可用

### 7.2 推荐目录与配置

- `config/app.toml`
- `storage/uploads`
- `storage/outputs`
- `storage/temp`

### 7.3 依赖校验方式

- 启动前检查端口是否被占用
- 检查数据库和 Redis 连接串是否可用
- 检查 FFmpeg 是否可执行
- 检查存储目录是否存在且可写
- 检查模型配置是否完整

## 8. 上线前检查清单

- 核心上传与生成主链路通过
- 任务状态流转完整
- 生成数量和时长范围校验通过
- 片头片尾拼接结果可播放
- 失败重试逻辑已验证
- 日志链路可查询，事件顺序和内容与实际执行一致
- 配置校验逻辑已验证
- 本地依赖项已确认可用
- 结果素材可预览、可下载、可追踪
- 关键错误信息清晰且可定位
- 回归样本已固定，便于后续版本对比

## 9. 通过标准

首版测试通过的最低标准：

- 核心业务链路可跑通
- 所有关键验收用例通过
- 重大错误有明确提示，不会静默失败
- 结果素材满足基本投放要求
- 配置和环境依赖问题可在启动阶段被发现
